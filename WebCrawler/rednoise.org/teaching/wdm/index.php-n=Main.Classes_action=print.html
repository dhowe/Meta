Classes (and Objects)

In Processing (and other 'object-oriented' languages, e.g. Java, C++, Perl, Python, etc.) a class is a 'blueprint' or 'template' for the creation of objects. A class can be thought of as a combination of data (variables) and behaviors (functions). To write (or declare) a class, we supply the following information:

  • Name of the class -- this is done by saying "class ClassName". As always, we then enclose the code for the class in curly brackets. As our example, we will create class called 'Car'. 
         class Car
         {
            // the guts
         }
  • A constructor -- The constructor is a function that always has the same name as the class. It specifies any required parameters for creating an 'instance' of the object. (Note that a class can have more than one constructor, each specifying different ways of creating the class). To use or 'call' a constructor, the 'new' keyword is used: 
         Car myCar = new Car("zip", 50, 100, 200, 7.8);
  • The data variables for the class -- these are variables that will hold the data associated with each instance (or object) created from our template. This is done in the same way as described in variables; e.g., by supplying the type, variable-name, and (optionally) and initial-value.

The Car class below has 5 variables 

         int color;
         int xpos;
         int ypos;
         float velocity;
         String model;
  • The functions for the class -- we can add functionality to our class by writing functions (often called 'methods'). This is done in the same way as described in functions; e.g., by supplying the function-name, return-type, parameters, and code.

The Car class below has just 2 functions (not including the constructor): 

         void drive() { ... }

            and

         void render() { ... }

Here's the full code for our 'Car' class: 

  class Car
  {

    // DATA -------------------------------------

    int color;
    int xpos;
    int ypos;
    String model;
    float velocity;

    // CONSTRUCTOR ------------------------------

    Car(String mod, int col, int x, int y, float vel) {
      model = mod;
      color = col;
      xpos = x;
      ypos = y;
      velocity = vel;
    }

    // FUNCTIONS --------------------------------

    void render () 
    {
      fill(color);
      rect(xpos, ypos, 20, 10);
    }

    void drive () 
    {
      xpos = xpos + velocity;
      if (xpos > width) { 
        xpos = 0; 
      }
    }

  } // END

From the definition above, it should be clear that the variables and functions defined in our Car class apply to specific instances (objects) of the Car class that we created. So, for instance, we can create 2 car objects as follows: 

       Car car1 = new Car("zip", 50, 100, 200, 7.8);

       Car car2 = new Car("zap", 20, 80, 100, 5.8);

Since we created these with different values in their constructor, they will have different values for their variables (car1 is faster than car2, for example). Similarly if we want to change the velocity of car2, we can change that variable and it has no affect on car1: 

       car2.velocity = 10.4;

But what about cases where we want to keep track of something for the whole class? For example, lets say we want to track the # of Car objects we've created so far. We wouldn't want to store this info in every Car object, it would always be the same and that could be a lot of wasted space if we had thousands or millions of cars. Instead, most OO languages have the notion of static variables and methods, which 'belong' to the Class itself, and can be thought of as being shared by all Car objects. So we might add a line like this to our Car class: 

       static int carCount = 0;

we could also add a static method to access this variable: 

       static int getCarCount() {
         return carCount;
       }

Now even other objects of other types can ask our Car class for its total count. Any idea how we might update the 'carCount' variable so it accurately reflects the # of created cars?

<ASIDE>

Getters', 'Setters', & 'Encapsulation

NOTE: In the code above, where we set the velocity: 

       car2.velocity = 10.4;

we used a dot (period) to specify that velocity belonged to the specific object called 'car2. This is often referred to as 'direct-access' (specifically via 'dot-notation'). When using libraries, we often find that some (or all) variables are not available via direct-access. This is because they are often made 'private' by library authors; that is, accessible only within their class (we'll talk about other 'visibility modifiers' like 'public' and 'protected' later on. Often, authors will instead provide methods allowing you to access such variables, generally referred to as 'getters' and 'setters', and these have (by convention) a specific form. For our 'velocity' variable, the getter and setter methods might look (respectively) like this: 

       float getVelocity() {
         return velocity;
       }
           AND

       void setVelocity(float newVel) {
         velocity = newVel;
       }

This technique is a type of 'encapsulation' that is generally considered good engineering practice, allowing the code in the setter/getter functions to verify, for example, that its OK to modify that variable at that moment, or that the value supplied is in an acceptable range. In less formal contexts (like 'art') one tends to see both forms (direct-access, [via dot-notation], and getter/setter style), so if a variable is not accessible directly, be sure to check for a method that allows you to read and/or modify it.

Processing is a bit of strange case in that it contains a mix of both styles (though rarely using the conventions described above). For example, to determine the frame-rate of a sketch one can directly access the 'frameRate' variable, while to specify a desired frame-rate, we must call the method frameRate() specifying the new value as an argument, e.g., frameRate(24). In this case its unclear (without examining the source-code) exactly what would happen if we were to set the 'frameRate' variable directly. Any guesses?

</ASIDE>
